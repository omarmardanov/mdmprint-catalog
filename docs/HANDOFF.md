# Передача в разработку: перенос страницы каталога на WordPress

Кому: разработчику, который переносит прототип на текущий сайт mdmprint.ru.

**Что передаётся:** `prototypes/usluga.html` (4 500 строк, 370 КБ) + `assets/css/{tokens,kit,fonts}.css`
+ шрифты + картинки. Это **эталон вёрстки**, а не черновик: вся страница выверена на десктопе и
мобилке, числа в CSS — не «на глаз», а из дизайн-системы.

**Сопутствующие документы** (читать все, они не дублируют друг друга):

| Файл | О чём |
|---|---|
| `DESIGN_SYSTEM.md` | правила: токены, шкалы, поверхности, формы, тач. **Источник правды — код** |
| `TEMPLATES.md` | карта блоков: один шаблон, три типа записи |
| `CMS_FIELDS.md` | админка: поля, лимиты, пустые и крайние значения |
| `QUESTIONS.md` | что нужно узнать до старта (ХП API, лиды, юрлинки) |
| `IA_catalog.md` | информационная архитектура каталога |
| `DEV_WALKTHROUGH.md` | **прочитать первым**: я прошёл эту задачу как разработчик и записал 9 мест, где встаёшь — включая 3 мины, которые снаружи не видно |

## 1. Главное правило переноса

**Не «перевёрстывать по макету», а перенести код.** Прототип — не картинка, он уже решает десятки
неочевидных вещей: тач-цели, залипающий hover, анти-зум iOS, обрезка ленты по краю экрана,
специфичность селекторов. Каждое из этих мест ловилось замером и стоило времени; переписывание «как
понял» вернёт их обратно.

Что можно менять свободно: разметку **под шаблонизатор** (циклы, подстановки полей).
Что менять нельзя без разговора: значения в `tokens.css`, компоненты `kit.css`, брейкпоинт 900px.

---

## 2. Порядок работ

1. **Подключить ассеты**: `fonts.css` → `tokens.css` → `kit.css`, именно в этом порядке.
   **Обязательно с `?v=ГГГГММДД`** и бампать при каждой правке CSS. Без версии браузер отдаёт старый
   файл часами: 16.07 на этом потеряли час — телефон показывал новый HTML со старым CSS, и это
   выглядело как «шрифты внезапно выросли».
2. **Завести типы записей, таксономии и поля** по `CMS_FIELDS.md` §1–2.
3. **Перенести существующий контент** в поля. Тут вылезут **лимиты**: часть текстов длиннее нормы,
   и это не повод лимиты снять — это повод сократить тексты (см. §10 ДС, там объяснено, что ломается).
4. **Собрать шаблон** из блоков; каждый блок — отдельный partial, включается по типу записи и по
   наличию данных.
5. **Подшапка-навигация собирается из включённых блоков**, а не из статичного списка.
6. **Интеграции**: ХП API (цены), SmartWidgets (отзывы), приём лидов из форм.
7. **Приёмка** по чек-листу §7.

## 4. Интеграции

### 4.1. ХП (CRM/ERP) — цены

Правда о ценах живёт в ХП. Ручную таблицу **не заводим**: менеджер предупредила — «цены меняем, а про
табличку никто не вспомнит; в сезон нередко делаем несколько повышений». Google-таблица из
обсуждения — временный артефакт, на сайт её тянуть не надо.

Модель совпадает у сайта и у ХП: **тиснение — это опция калькулятора продукции**, а не отдельный
калькулятор. Строка прайса = изделие + свойство. Калькулятор один на сайт, страница передаёт в него
предустановку параметром.

Что запросить у ХП — в `QUESTIONS.md` §1. Ключевое: **«от 18 ₽/шт» — от какого тиража** (приладка
размазывается по тиражу, цена за штуку от него сильно зависит), НДС, срок, дата обновления цены.

Поведение при недоступном API: показать последнюю известную цену с датой **или** скрыть цифры и
вести в калькулятор. Решить до разработки, а не в проде.

### 4.2. SmartWidgets — отзывы

Виджет рендерит свой DOM в скрытый `.sw-app`; берём только данные из `window.swapp_data` после
события `swAppComplite` и рисуем своей разметкой. **Не подключать их свайпер и стили.**

Ограничение: фильтра по услуге нет — на всех страницах одни и те же отзывы компании.

**Грабля:** виджет **заменяет** статичные карточки (`grid.replaceChildren`). Любая разметка,
навешенная при `load`, уходит вместе с ними — вешать после рендера виджета (в коде это уже так).

### 4.3. Формы → лиды

Куда уходят заявки — **вопрос заказчику** (`QUESTIONS.md` §2). Honeypot `company` уже есть,
капча не нужна. Согласие на ПД обязательно, ссылки на политику сейчас заглушки.

### 4.4. Аналитика — размечать ДО сборки шаблона

В прототипе разметки аналитики **нет вообще**: ни счётчиков, ни событий, ни атрибутов-хуков.
Это осознанный пробел прототипа, а не забытая работа, — но закрывать его надо **до** сборки
шаблона, а не после. Причина простая: половина конверсионных действий на странице это не переходы,
а клики и отправки форм, и если хуки не заложены в разметку сразу, потом их придётся вешать по
CSS-селекторам, которые поедут при первой же правке вёрстки.

**Что обязательно померить** (все элементы в прототипе уже есть):

| Действие | Где |
|---|---|
| Отправка «Оставить заявку на расчёт» | модалка-лид, две точки входа: герой и блок цен |
| Отправка «Перезвоним» | модалка звонка из шапки |
| Отправка вопроса в ЧАВО | форма в блоке вопросов |
| Клик по телефону | шапка (десктоп), контакт-ряд (мобилка), футер |
| Клик по мессенджерам | Телеграм / Макс / почта — те же три места |
| Переход в калькулятор | блок цен и меню; отдельная цель — там скидка −5% |
| Клик «Заказать образец» | герой |
| Скачивание файлов | шаблоны и требования, PDF/AI |
| Подписка на рассылку | футер |
| Клик по соцсетям | футер |

**Решения, которые должен принять заказчик, а не разработчик** (вынесено в `QUESTIONS.md` §6):
какой счётчик и чей аккаунт, ставить через GTM или в тему напрямую, включать ли вебвизор и карту
кликов (это запись поведения — вопрос к политике ПД, раз на странице есть формы), нужна ли
электронная торговля, раз в шапке живёт корзина.

**Требование к разработчику:** события вешать на **атрибуты в разметке** (`data-*`), а не на классы
оформления. Классы принадлежат дизайн-системе и меняются при редизайне; аналитика, привязанная к
ним, тихо умирает, и заметят это через квартал по провалу в отчётах.

## 5. Технические требования

| Тема | Требование |
|---|---|
| Брейкпоинт | один: `900px`. Мобильная типошкала и `--gap-card` переопределяются в `tokens.css` |
| Шрифты | AttractiveCond (заголовки) + Roboto (текст), self-host в `assets/fonts` (1.1 МБ). Не подключать Google Fonts |
| Картинки | `.webp`, `loading="lazy"`, у всех `width`/`height` — без них поедет CLS |
| Прозрачные PNG | **нельзя** с `mix-blend-mode:darken` — у прозрачных пикселей RGB=0, зальёт чёрным |
| CSS | не инлайнить и не минифицировать в один файл с чужим CSS темы: специфичность прототипа рассчитана на порядок `tokens → kit → страница` |
| JS | ванильный, без зависимостей. jQuery темы не мешает, но и не нужен |
| Доступность | `prefers-reduced-motion` учтён; `:focus-visible` у всех интерактивов; тач-цели ≥44px; аккордеоны нативные (`<details>`) — не заменять на div+JS |
| Печать/SEO | H1 один; уровни заголовков не прыгают; крошки — микроразметка на стороне WP |

**Специфичность — главная мина.** В прототипе шесть мест, где правило написано и молча не работает
(`.usl section` (0,1,1) бьёт `.usl-price` (0,1,0); `:not([class])` даёт +1; `body.v-after .X` из
более позднего `<style>` бьёт дважды). При переносе в тему добавится ещё и её CSS. **Проверять
замером** (`getComputedStyle`), а не чтением кода.

## 7. Чек-лист приёмки

**Вёрстка**
- [ ] На 390 нет горизонтального скролла ни на одной секции.
- [ ] Тач-цели ≥44px: кнопки, ссылки-источники, чекбоксы, аккордеоны.
- [ ] `:hover` не залипает после тапа (все hover — внутри `@media (hover:hover)`).
- [ ] Вспышка тапа погашена, но у каждого интерактива есть `:active`.
- [ ] Поля ввода 16px — iOS не зумит при фокусе.
- [ ] Ленты обрезаются по краю ЭКРАНА, текст остаётся на сетке 16px.

**Формы и модалки** (добавлено 20.07)
- [ ] На мобилке модалка — шторка снизу: кнопка отправки видна **всегда**, крестик не уезжает.
- [ ] Кольцо фокуса у полей не срезано боковой границей скролл-контейнера.
- [ ] Кнопка отправки **не заблокирована**: по нажатию подсвечиваются все проблемные поля,
      прокрутка уводит к первому, включая чекбокс согласия.
- [ ] Ошибка появляется у `select` и `textarea`, а не только у `input`.
- [ ] Пустое поле не краснеет от простого ухода фокуса.
- [ ] Второй выбранный файл ДОБАВЛЯЕТСЯ к первому, а не заменяет его; лишний удаляется.
- [ ] Заголовок модалки склонён верно: «Заявка на ламинаци**ю**», а не «на ламинация».
- [ ] Липкий CTA не показывается, когда на экране есть настоящая кнопка заявки или футер.

**Данные**
- [ ] Пустой блок скрыт вместе с пунктом подшапки.
- [ ] Крайние значения (0 / 1 / максимум) проверены рендером.
- [ ] Цены тянутся из ХП; при недоступности API поведение осознанное.
- [ ] Лимиты полей стоят в админке со счётчиком.

**Дизайн-система**
- [ ] Ни одного хардкода кегля/цвета/отступа — всё из токенов.
- [ ] Ни одного дубля китового класса.
- [ ] `?v=` у CSS бампается при правках.
