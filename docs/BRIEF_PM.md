---

## 1. Что происходит в двух абзацах

У mdmprint.ru **уже есть** страницы каталога. Мы их не создаём с нуля — мы меняем им дизайн.
Это миграция контента в новый шаблон, а не наполнение пустого сайта: старые URL надо
сохранить, редиректы не плодить.

Дизайн сдан не макетом, а **свёрстанным прототипом**: `prototypes/usluga.html`, 4 500 строк,
десктоп и мобилка, живые интерактивы. Это принципиально: страница уже решает десятки неочевидных
вещей — тач-цели, залипающий hover на тачскрине, анти-зум iOS, обрезка лент по краю экрана,
специфичность селекторов. Каждое из этих мест ловилось замером. **Задача разработки — перенести
код, а не перевёрстывать по картинке.** Переписывание «как понял» вернёт эти баги обратно, и
поймают их уже в проде.

## 2. Состав редизайна — компоненты

Редизайн — это не одна страница. Ниже — всё, что входит; PM режет на эпики отсюда. Большинство
свёрстано в эталоне `usluga.html` (шапка, футер, формы, страница каталога — в одном файле),
отдельные стенды указаны.

| Компонент | Что это | Прототип / док | Тип |
|---|---|---|---|
| **Шапка** | мега-меню (десктоп), drawer drill-down (мобилка), поиск с подсказками | `usluga.html` — единственный источник | сквозной |
| **Футер** | навигация, контакты, соцсети, форма рассылки | `usluga.html` | сквозной |
| **Формы** | заявка, звонок, вопрос: модалки-шторки на мобилке, валидация, файлы, контакты М2 (телефон обязателен, e-mail нет) | `usluga.html` (`setupLead` + ЧАВО) | сквозной |
| **Каталог: страница** | гибкий шаблон технология/материал, ~14 блоков | `usluga.html` (эталон) | шаблон, детали в §3 |
| **Верхний блок «Технологии печати»** | табы групп + плитки услуг; меняется только верх страницы | `tehnologii-top.html`, `TEHNOLOGII_TOP.md` | отдельная страница |
| **Верхний блок «Каталог материалов»** | тот же паттерн, что Технологии | — (ещё не свёрстан) | отдельная страница |
| **Сквозное** | липкий CTA (мобилка), разметка аналитики, версионирование CSS `?v=` | `usluga.html` | сквозное |

**Шапка, футер и формы — сквозные:** делаются один раз, живут на всех страницах. Их логично
вынести в отдельные эпики раньше страничных.

## 3. Каталог: страница (гибкий шаблон)

**Один гибкий шаблон обслуживает ДВА типа записи**: технология (она же услуга — как назвать,
пока не решено) и материал. Блоки включаются по типу и по наличию данных — не «два шаблона»,
а «один гибкий»: дешевле в разработке, но требует аккуратности с условиями вывода.

| Что | Объём |
|---|---|
| Шаблон страницы | 1 гибкий (технология + материал) |
| Блоков в шаблоне | ~14, каждый — отдельный partial |
| Полей в админке | см. `CMS_FIELDS.md`, с лимитами символов |
| Интеграций | 3: ХП (цены), SmartWidgets (отзывы), приём лидов из форм |
| Аналитика | 10 действий под замер, разметки сейчас нет |
| Страниц к миграции | все существующие страницы каталога этих типов |

**Не входит в этот объём** (отдельные задачи, если понадобятся):
страница онлайн-калькулятора — **её не существует**. Пока её нет, блок калькулятора на странице
**скрыт** флагом «есть URL» (обычный условный вывод, отдельного дизайна не нужно); появится
страница → редактор вписывает URL → блок включается. Блок «Характеристики» для шаблона
материалов — собран, но не согласован.

## 4. Состав работ

1. Подключить ассеты (`fonts` → `tokens` → `kit`, строго в этом порядке, с версионированием).
2. Завести типы записей, таксономии и поля по `CMS_FIELDS.md`.
3. Перенести существующий контент в поля.
4. Собрать шаблон из блоков-partial'ов, включение по типу записи и по наличию данных.
5. Подшапка-навигация — **собирается из включённых блоков**, а не статичным списком.
6. Интеграции: ХП API, SmartWidgets, лиды.
7. Разметка аналитики (закладывать на шаге 4, не после).
8. Приёмка по чек-листу `HANDOFF.md` §7.

Шаг 3 стабильно недооценивают: при переносе вылезут тексты **длиннее лимитов**. Это не повод снять
лимиты — это повод сократить тексты. Заложить время редактора, а не только разработчика.

## 5. Что блокирует старт

Ответы нужны **до** первой строки кода, иначе работа переделывается.

| Блокер | Кому вопрос | Где подробно |
|---|---|---|
| API ХП: доступ, формат, есть ли минимальный тираж отдельным полем | разработчик ХП | `QUESTIONS.md` §1 п. 1–8 |
| «от 18 ₽/шт» — от какого тиража, НДС, округление | менеджер | `QUESTIONS.md` §1 п. 9–13 |
| Поведение при недоступном API ХП | директор | `QUESTIONS.md` §1 п. 14 |
| Страница калькулятора: когда будет и какой URL | директор | `QUESTIONS.md` §1 п. 15 |
| Куда уходят заявки из форм | директор | `QUESTIONS.md` §2 |
| Счётчик, GTM или напрямую, вебвизор | директор + маркетолог | `QUESTIONS.md` §6 |

Ещё две вещи **меняют объём работ**, поэтому решать их лучше до планирования, а не в середине
спринта: свести три почти одинаковые формы в один компонент (`QUESTIONS.md` п. 30) и определиться
со смыслом блока «Где применяется» (п. 31).

## 6. Риски

**Специфичность CSS — главная техническая мина.** В прототипе шесть мест, где правило написано и
молча не работает, потому что его перебивает более специфичный селектор. При переносе в тему
WordPress добавится ещё и её собственный CSS. Проверять **замером** (`getComputedStyle`), а не
чтением кода. Заложить время на это — симптом выглядит как «дизайн поехал», а причина не видна.

**Виджет отзывов перерисовывает DOM.** SmartWidgets заменяет карточки целиком; любая логика,
навешенная при загрузке страницы, уходит вместе с ними. В прототипе это решено, но при переносе
легко сломать заново.

**Юридический риск в ассетах.** Логотипы Яндекс.Карт и 2ГИС, бейдж «Хорошее место» сейчас
нарисованы нами. Нужны официальные, иначе это использование чужих товарных знаков.

**Кеш CSS.** Без `?v=` браузер отдаёт старый файл часами. На этом уже терялось время: телефон
показывал новый HTML со старым CSS, и это выглядело как «шрифты внезапно выросли».

**561 заглушка `href="#"`** — ссылки каталога, статей, политик. Подставить реальные URL.
Это не строчка работы, это отдельная ревизия.

## 7. Что читать и в каком порядке

Всё в репозитории, точка входа — `index.html`.

1. `DEV_WALKTHROUGH.md` — **разработчику первым**: 9 мест, где встаёшь, включая 3 мины.
2. `HANDOFF.md` — состав передачи, порядок работ, что живое и что заглушка, чек-лист приёмки.
3. `QUESTIONS.md` — открытые вопросы по адресатам, помечено что блокирует.
4. `TEMPLATES.md` — карта блоков.
5. `CMS_FIELDS.md` — поля админки с лимитами.
6. `DESIGN_SYSTEM.md` — правила ДС. Источник правды — код, не макет.

## 8. Чего нельзя менять без разговора с дизайном

- значения в `tokens.css` и компоненты `kit.css`;
- брейкпоинт 900px;
- лимиты символов в полях;
- замену нативных аккордеонов (`<details>`) на `div` + JS.

Свободно меняется разметка **под шаблонизатор**: циклы, подстановки полей, вынос в partial'ы.
