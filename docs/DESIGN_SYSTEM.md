# MDMprint — Дизайн-система (v2, по коду)

**Источник правды — код, а не этот файл.** Значения живут в `assets/css/tokens.css`, компоненты —
в `assets/css/kit.css`. Документ описывает **правила и решения**: почему так, где граница, что
считать нарушением. Числа дублируются здесь только для чтения — при расхождении прав код.

> **История.** v1 (2026-06-16) была предложением под Figma и к июлю разошлась с реальностью почти
> во всём: бренд `#f7a705` (в коде `#FFC100`), заголовки Roboto Condensed (в коде AttractiveCond —
> шрифт нашёлся и лежит в `assets/fonts/`), шкала H1 34 / H2 26 (в коде 40 / 32), кнопки S36/M44/L52
> (в коде lg48 / md36 / sm32), контейнер 1156 (в коде 1176). Две правды хуже одной: v2 переписана
> по факту. Раздел про Figma-файл и хендофф (§11) из v1 сохранён — он не устарел.

Подключение: `fonts.css` → `tokens.css` → `kit.css`. **В ссылках обязателен `?v=ГГГГММДД`** и
бампать его при каждой правке `assets/css/*`: CSS лежит по неизменному адресу, браузер держит его
часами. 16.07 на этом попались — телефон показывал новый HTML со старым CSS, и выглядело это как
«шрифты внезапно выросли».

## 1. Правило нарушений

Дизайн-система — это не список значений, а **договор о том, где значение брать**. Практика показала
три типовых нарушения; они и есть чек-лист ревью:

1. **Хардкод.** Число вместо токена. Проверяется: `grep -o "font-size:[0-9]*px"`. Опасно не тем, что
   «некрасиво», а тем, что при правке токена такое место за ним не поедет: 16.07 в прототипе нашлось
   35 хардкодов кегля, среди них `font-size:19px` — кегля 19 в шкале нет вовсе, и заголовок был мельче
   равного себе по уровню.
2. **Дубль китового класса.** Свой класс, посимвольно повторяющий существующий (`.art-row__meta` =
   `.mk-eyebrow`; `.usl-subnav a` = `.mk-tab`). Перед новым классом — грепнуть кит.
3. **Выбор по месту.** Форма/цвет/отступ выбраны «как тут смотрится», а не по правилу. Так на странице
   завелись три языка форм иконок сразу (см. §5).

**Проверять замером, а не чтением CSS.** Правило может быть написано и молча не работать: за проект
поймано шесть случаев, где селектор проигрывал по специфичности или порядку (`.usl section`
(0,1,1) бьёт `.usl-price` (0,1,0); `:not([class])` даёт +1; `body.v-after .X` из более позднего
`<style>` бьёт `.X` дважды). Диагностика — `--dump-dom` + `getComputedStyle`, не скриншот.

## 2. Цвет — роли, а не значения

Полный список — в `tokens.css`. Здесь только правила.

**Единая тёплая нейтральная шкала.** Все светлые нейтрали тёплые (беж/песок), холодного серого в
системе нет — он спорит с тёплым брендом `#FFC100`.

**Правило поверхностей** (нарушается чаще всего):

| Токен | Роль |
|---|---|
| `--surface-subtle` | фон **секции** |
| `--surface-gray` | подложка **карточки** |
| `--surface-card` | **белая плашка** поверх нейтрали |
| `--surface-gray-2` | hover карточки |
| `--surface-on-light` / `--surface-on-dark` | тональная заливка **поверх** светлого / тёмного |

Отсюда правило: **серая карточка живёт на белой секции.** Если секция стала бежевой, серая карточка
на ней мутнеет — значит либо белая плашка, либо без заливки вовсе.

**Тёмная поверхность = акцент, один на страницу.** Два тёмных блока — уже не акцент, а тяжёлый низ.
Поэтому «Готовые материалы» тёмными не делаем, хотя соседний калькулятор тёмный.

**Правило одного акцента.** Жёлтый — только главное действие и активное состояние. Жёлтый на
вторичном переключателе (просьба маркетолога, 16.07) отклонён: акцентный цвет на вторичном действии
обесценивает акцент. Бренд как **цвет текста** — только `--brand-text` (`#A87D00`): сам `#FFC100`
на светлом даёт 1.7:1 и нечитаем.

**Чужие бренд-цвета** (`--src-yandex`, `--src-2gis`) подменять нельзя.

## 3. Типографика

Заголовки — `AttractiveCond`, КАПС. Текст и UI — `Roboto`.

**Трекинг в Figma задан в ПРОЦЕНТАХ**, а MCP отдаёт голое число: `letterSpacing: 2` у H4 = 2% =
0.36px при 18px. В токенах поэтому `em`. Проглядеть это — значит разрядить буквы в 3–6 раз.

### Десктоп

| Ступень | Size / Line | Трекинг |
|---|---|---|
| `--t-display` | 50 / 54 | .01em |
| `--t-h1` | 40 / 44 | .01em |
| `--t-h2` | 32 / 44 | .015em |
| `--t-h3` | 24 / 28 | .02em |
| `--t-h4` | 18 / 22 | .02em, w600 |
| `--t-lead` | 18 / 27 | — |
| `--t-body` | 16 / 24 | — |
| `--t-ui` | 15 / 20 | строки меню, поля ввода, строки-заголовки |
| `--t-button` | 15 / 16 | w500 |
| `--t-small` | 14 / 20 | — |
| `--t-caption` | 13 / 16 | — |
| `--t-label` | 12 / 16 | .04em, КАПС |
| `--t-micro` | 11 / 14 | .04em — длительность, вес файла, бейджи |

`--t-ui` и `--t-button` совпали кеглем — это **совпадение, а не синоним**: у кнопки свой леадинг под
высоту 48 и свой вес. Держим врозь, иначе правка кнопки поедет по всему меню.

`--t-label` набирается **только капсом**, потому трекинг .04em: макетные 0.5% при 12px дают 0.06px,
то есть трекинга нет, и прописные слипаются.

### Мобильная (≤900px)

Переопределяем **сами ступени** в `tokens.css`, а не заводим вторые токены (`--t-h1-m`): тогда всё,
что набрано по ДС, съезжает на мобильную лестницу само, и в вёрстке не надо дублировать правила под
брейкпоинт.

| Ступень | Десктоп | Мобилка |
|---|---|---|
| `--t-display` | 50 / 54 | 40 / 44 |
| `--t-h1` | 40 / 44 | **34 / 38** |
| `--t-h2` | 32 / 44 | **24 / 28** |
| `--t-h3` | 24 / 28 | **18 / 22** |
| `--t-h4` | 18 / 22 | **16 / 20** |
| `--t-lead` | 18 / 27 | **16 / 24** |

Четыре различимых уровня — обязательны. До 16.07 h3 и h4 на мобилке оба падали в 18 и различались
только весом (700 против 600) — этого мало, иерархия не читалась.

Текстовые ступени (body / small / caption / label / micro) на мобилке **не меняются**: 16px — порог,
ниже которого iOS зумит поле при фокусе, а мера строки чинится шириной контейнера, не кеглем.

### Правила набора

- **Один H1 на страницу**, дальше по уровням без прыжков.
- **Контраст минимум по двум признакам из трёх** (кегль / вес / цвет). Одного веса мало.
- **Леадинг по роли, а не по кеглю**: 1.5 — интерлиньяж сплошного текста. Заголовок карточки в
  1–2 строки от него расползается — там 1.25 (`.mk-mcard__title` = 16/20).
- **Мера строки**: десктоп 45–75 знаков; мобилка 35–50. Заголовки секций — `max-width:27ch` +
  `text-wrap:balance` (замер на 390: 1ch ≈ 11.3px).
- **Класс задаёт вид, тег — структуру.** `<h3 class="mk-h4">` законно: уровень в документе и ступень
  набора — разные вещи.

## 4. Сетка, отступы, ритм

- Контейнер `--container: 1176`, поля `--container-pad: 24`.
- **Модульная сетка: 12 колонок × 76px, гаттер 24.** В CSS не объявлена токеном — зашита в `fr`:
  `.price-grid{776fr 376fr}` = 8+4, `.vidy-grid{repeat(3,1fr)}` = 4+4+4, `.adv-grid{1fr 1fr}` = 6+6.
  Легко проглядеть (я проглядел — заказчик поправил 16.07).
- **На мобилку сетка не переносится**: те же 12 колонок при контейнере 358 дают колонку 7.8px.
  Мобильная раскладка держится на полях 16px и 1–2 колонках. Мера текста на колонки не садится —
  это разные системы: колонки про раскладку блоков, мера про длину строки.
- Шкала отступов `--sp-2 … --sp-80`, вертикаль секции `--section-y: 80`.
- **Правило внешних и внутренних**: расстояние между группами обязано быть заметно больше, чем внутри
  группы (12 внутри → 32 между). Одинаковые зазоры не группируют. Когда фон убран, границу групп
  держит только расстояние — либо линейка (см. §7).

### Паддинг — лестница по размеру, а не по вкусу

| Роль | Десктоп | Мобилка | Кто |
|---|---|---|---|
| Крупная карточка-контейнер (≥300px, внутри несколько групп) | **24** | **16** | прайс, форма, файлы, отзыв |
| Плитка (150–300px) | **16** | **16** | `.mk-tile` |
| Тело плитки под кадром | **16** | **12** | `.mk-mcard__body` |
| Строка списка | **12** | **12** | `.mk-filerow`, `.mk-pcard` |

**На мобилке паддинг крупной карточки = полю контейнера (16).** Не совпадение: тогда текст внутри
карточки стоит на той же сетке, что текст вне её, и карточки не спорят с полосами во всю ширину
экрана. Десктопные 24 на 390 съедали бы 13% ширины.

Исключения, которые исключениями не являются: у блока, где карточка **разобрана** на строки (фон
снят — «Цены», «Требования» на мобилке), паддинга нет вовсе — это уже не карточка. У полосы во всю
ширину экрана горизонталь равна полю контейнера по определению.

До 17.07 правила не было, и одна роль набиралась тремя числами: 28 (прайс, требования), 24 (файлы,
форма), 20 (отзыв) — каждый блок решал сам. Числа 28, 20, 10 и асимметрия `16/20/20` из шкалы
выпали: паддинг — часть компонента, и живёт в ките (`.mk-review`, `.mk-mcard__body`, `.mk-filerow`),
а не в правилах страницы.

### Зазор между карточками — токен `--gap-card`

`--gap-card`: **24** на десктопе, **12** на мобилке (`tokens.css`, там же где мобильная типошкала).

**Правило: зазор между плитками ≤ поля контейнера.** Иначе плитки стоят друг от друга дальше, чем
от края экрана, сетка рассыпается на отдельные объекты и читается как «нет сетки». На мобилке поле
16 → зазор 12. Проверяется замером: для каждой сетки сравнить `column-gap` с `(VP − ширина .wrap)/2`.

Почему токен, а не число по месту: правило держится, только если зазор и поля меняются согласованно.
До 16.07 каждая сетка писала свой `--sp-24`/`--sp-20`, и ленты жили по разным законам — кит давал 20,
отзывы переопределяли на 24. **Ленты обязаны считать ширину карточки от того же токена**:
`--item-w: calc((100% - 1.2 * var(--gap-card)) / 2.2)` — иначе «2.2 карточки в кадре» разъедется,
как только зазор изменится.

**Число колонок — тоже часть правила.** Зазор не спасёт, если колонок больше, чем позволяет экран:
«Сочетается» держал 3 в ряд и на мобилке — плитка 103px на 390.

Число колонок выбирается **по содержимому карточки**, а не одинаково для всех:

| Что в карточке | Колонок на мобилке | Пример |
|---|---|---|
| Проза (заголовок + описание) | лента 80% (см. §7) | «Виды тиснения» |
| Заголовок + иконка/картинка | **2** | «Сочетается» — внутри текст рядом с картинкой |
| Только короткая подпись под кадром | **3** | «Где применяется» — плитка 111px, подпись в 2 строки |

Проверка — не «влезло ли», а **влезло ли самое длинное слово**: измеряется canvas-ом
(`ctx.measureText`), а не на глаз. Если слово шире колонки, кегль не спасает — нужны переносы
(`hyphens:auto` + `lang="ru"` у `<html>`) или редакторский лимит (§10).

## 5. Радиусы и формы

`--radius-inside: 4` · `--radius-md: 8` · `--radius-outside: 14` · `--radius-pill: 999`

- Карточки и тайлы — `--radius-outside` (14).
- Кнопки — пилюля; на странице услуги включён `body.shape-rounded` → 8.
- **Правило форм иконок (16.07):** контейнер с **глифом** — скруглённый квадрат (`--radius-md` от
  26px; `--radius-inside` у бейджей ≤24: 8 на 22px даёт уже не квадрат, а кляксу). **Круг — только
  там, где круг и есть суть элемента**: точки-маркеры списков (5–8px) и аватар в отзывах.
  Скруглённый квадрат — ещё и у картинки-сущности (логотип, превью, жёлтый тайл каталога).

До этого форма выбиралась по месту: шесть кружков, квадрат в футере, жёлтый тайл в каталоге — три
языка без правила. Заодно `50%` и `999px` больше не спорят как два способа сказать одно.

## 6. Кнопки

Высоты: `lg 48` (контент) · `md 36` (шапка) · `sm 32` (топбар). Форма — пилюля.

| Вид | Что | Где |
|---|---|---|
| `--primary` | жёлтая, текст `--on-brand` | главное действие, одно на зону |
| `--dark` | чёрная | шапка, промо |
| `--ghost` | белая + рамка | **вторичная на светлом** |
| `--ghost-inverse` | контур на тёмном | вторичная на тёмном |
| `--soft-inverse` | тональная на тёмном | тише solid |
| `--inverse` | белая сплошная на тёмном | промо-CTA |
| `--link` | текст с подчёркиванием | третичное |

- **Светлого `--soft` нет намеренно**: `--surface-on-light` поверх белого даёт `#f5f4f1`, а
  disabled — `#f5f3ed`; вторичная кнопка была неотличима от выключенной. Вторичная на светлом —
  `ghost`: контур и есть affordance.
- **`disabled` — плоский серый**, не прозрачность: полупрозрачный жёлтый читается как активная.
- **`.mk-btn` привязан к поверхности.** `--soft-inverse`, `--ghost-inverse`, `.mk-card--dark`
  работают только на тёмном. Класс на брейкпоинте не подменить — если блок теряет тёмный фон на
  мобилке, его кнопка переопределяется на светлый эквивалент.

### Стрелка в кнопке = уход со страницы

Со стрелкой (`.mk-arr`): переход на другую страницу («Читать», «Смотреть», «Онлайн-калькулятор»).
Без стрелки: действие на месте — открыть модалку, отправить форму, раскрыть каталог. **Тег значения
не имеет**: «Оставить заявку» — это `<a href="#">`, но открывает модалку, значит стрелки нет.

Правило не выдумано: оно описывает сложившуюся практику (проверено по всем кнопкам разом, 16.07);
единственным исключением был калькулятор — исправлен.

## 7. Мобильные паттерны (приняты 15–16.07)

- **Полосы во всю ширину экрана вместо контейнера** — статбар, крошки, подшапка, ленты, строки цен и
  требований. Механика: `margin-inline: calc(var(--container-pad) * -1)` + собственный
  `padding-inline: var(--container-pad)` → фон и линии идут от края до края, текст остаётся на сетке
  16px. Для флекс-лент добавить `scroll-padding-left` и хвост `::after{flex:0 0 1px}`
  (`padding-inline-end` в скролл-контейнере игнорируется).
- **Ленты вместо колонок** для карточек с прозой: 2 колонки при 167px = мера 19 знаков, текст в
  лесенку. Карточка 80% ширины — соседняя подглядывает, скролл считывается без стрелок.
- **Плашки → строки.** Вложенность (секция → карточка → плашки) лечится не отменой контейнера, а
  снятием заливок и радиусов внутри: границы держит разделитель. Слоёв один вместо трёх.
- **Граница групп говорит языком блока.** Если блок собран строками с линейками, между группами тоже
  линейка (воздух 32 с обеих сторон), а не дырка.
- **Главный CTA — во всю ширину.**
- `overflow-x: clip`, а не `hidden`: clip не создаёт скролл-контейнер, и `position:sticky` выживает.
- Смена `position: static → relative` **оживляет чужой `top`**, оставшийся от липкости (ловили: группа
  уезжала на 123px). Ставить `top:auto`.

### Тач

- Все `:hover` — внутри `@media (hover:hover)`. На тач-экране hover срабатывает от тапа и **залипает**
  после перехода. Обёртка честнее гашения: правила просто нет там, где нет мыши.
- У каждого интерактива — свой `:active`, иначе тап без отклика.
- `-webkit-tap-highlight-color: transparent`: iOS рисует вспышку серым **прямоугольником**, игнорируя
  радиус. В ките гасим явным списком компонентов (чужую разметку кит трогать не вправе), на странице —
  одним правилом `a,button` (ручной список отставал: лого, крошки, согласие и весь футер).
- Тач-зона ≥ 44px (Apple) / ≥ 24px (WCAG AA).
- `input,select,textarea{font-size:16px!important}` — **константа платформы, не ступень шкалы**: ниже
  16 iOS зумит страницу при фокусе. Именно поэтому число, а не `var(--t-body)`.
- **Тач-зону можно добрать псевдоэлементом**, не раздувая сам элемент:
  `.btn{width:28px}` + `.btn::after{content:"";position:absolute;inset:-8px}` → 44px нажатия при
  28px рисунка. Нужно там, где кнопка сидит в плотной строке (крестик у файла в списке).

### Модалки на мобилке — шторка снизу (принято 20.07)

Центрированный попап на телефоне не работает: он тянется во весь экран, кнопка отправки уходит
за нижний край, по бокам остаются полосы оверлея. **Модалка ≤900px = шторка.**

- Приклеена к низу, во всю ширину, скруглена сверху, с ручкой.
- Высота — `max-height`, а НЕ фиксированная доля экрана: у формы звонка четыре поля, жёсткие
  70% дали бы ей полэкрана пустоты. Короткая форма остаётся короткой.
- **`dvh`, а не `vh`**: `vh` не учитывает съезжающую панель браузера.
- **Скроллит внутренний слой, а не сама шторка.** Тогда крестик остаётся на месте, а не уезжает
  с содержимым. Обязателен `min-height:0` на флекс-ребёнке, иначе он не сжимается и `overflow`
  не включается.
- **Кнопка отправки залипает** внизу (`position:sticky`) — видна и до, и после прокрутки.
  Фон обязателен, под ней проезжает текст. `env(safe-area-inset-bottom)` — иначе кнопка прячется
  под системную полосу жеста.
- **Боковые поля — на прокручиваемом слое, не на шторке.** Иначе поле упирается в границу
  скролл-контейнера и кольцо фокуса (`box-shadow` 3px) срезается.
- **Липкая шапка = сам заголовок.** Отдельный блок ради линии заводить не надо. Линия появляется
  **только при прокрутке**: постоянная граница разрезает заголовок с его же подзаголовком, и текст
  повисает между ними. Ширина 1px задана всегда, меняется только цвет — иначе раскладка дёргается.
- **Свайп вниз — только по верхней зоне** (~56px). Тянуть за содержимое нельзя: там свой скролл,
  и два жеста начинают спорить — отсюда рывки и залипание на полпути. Порог закрытия ~90px,
  короткий смах не должен закрывать заполненную форму.
- Стековый порядок: шторка ниже шапки и меню сайта; липкий CTA страницы — ниже модалки.

### Формы: обратная связь вместо заблокированной кнопки (принято 20.07)

**Кнопку отправки не блокируем.** Заблокированная кнопка сообщает «нельзя», но никогда не говорит
«почему», а сообщение об ошибке, живущее в обработчике отправки, у неё недостижимо — замкнутый круг.

- Ошибка формата — по **уходу** из поля, и только если поле не пустое: ругаться на первой букве
  адреса значит ругаться на того, кто ещё печатает.
- Ошибка «не заполнено» — по **нажатию** кнопки: метятся все проблемные поля разом, фокус и
  прокрутка уводят к первому. Внутри своего скролл-контейнера одного `focus()` мало.
- Ошибка гаснет **сразу**, как исправили.
- Подсветка обязана работать у `input`, `select` И `textarea` — правило только на `input` молча
  оставляет селекты без ошибки.
- Обязательный чекбокс согласия — тоже поле: подсвечивать и прокручивать к строке целиком,
  а не к квадратику 22px.
- **Не переусердствовать.** Имя: буква + два знака, цифры разрешены — «Отдел 5» законный ответ.

## 9. Тон и формулировки — ОТКРЫТО, нужно решение

Правила нет, и это видно. Факт по странице услуги (16.07, все кнопки разом):

| Подпись | Раз |
|---|---|
| «Оставить заявку на расчёт» | 3 |
| «Отправить заявку на расчёт» | 1 |
| «Оставить заявку» | 1 |
| «Отправить заявку» | 1 |
| «Заказать звонок» / «Жду звонка» | 1 / 1 |
| «Заказать образец» / «Заказать набор образцов» | 1 / 1 |

Одно и то же действие названо четырьмя способами. Предлагаемое правило (**на утверждение**):

1. **Глагол = что произойдёт от нажатия.** «Оставить заявку» — открывает форму. «Отправить заявку» —
   отправляет заполненную. Значит на странице до формы всегда «Оставить», внутри формы — «Отправить».
   Сейчас в блоке «Требования» стоит «Отправить заявку на расчёт», хотя кнопка **открывает** модалку.
2. **Одно действие — одна формулировка** на весь сайт. Выбрать канон: «Оставить заявку на расчёт».
3. **Хвост «на расчёт» — только там, где расчёт и есть цель.** В общем CTA он лишний.
4. **От первого лица только в подтверждении** («Жду звонка» в модалке — ок, это ответ пользователя).
5. **Без восклицаний, без «Вы» с большой буквы, без капса ради важности** — капс несёт ступень
   `--t-label`, а не эмоцию.

Решишь — впишу канон сюда и приведу страницу.

## 10. Редакторские ограничения (для CMS / переноса на WordPress)

**Договорённость с заказчиком (17.07): ни один блок, поле или текст не должен уехать в разработку без
ответа на вопрос «кто и как это администрирует».** Раздел ведём так, чтобы у разработчика и у
заказчика не оставалось вопросов, что откуда берётся и что можно менять. Для каждого блока
описываем: источник данных (статика / связь / выборка), где живёт каждое поле, лимиты, поведение
при переполнении и при нуле элементов.

### Два имени у сущности каталога

У группы/раздела каталога **по природе два разных имени**, и путать их нельзя:

| Имя | Где живёт | Пример |
|---|---|---|
| Навигационное | меню, URL, хлебные крошки | «Многостраничная полиграфия», «Виды карт», «По применению» |
| Клиентское (`short_name`, ≤20 знаков) | плитки, коммерческие блоки, текст | «Каталоги и брошюры» |

Навигационное имя оптимизировано под поиск по каталогу, клиентское — под чтение в плитке. В меню
первое читается нормально, в плитке — дико.

**Правило: клиентское имя живёт у САМОЙ группы, а не у страницы.** В WordPress — отдельное поле
таксономии/CPT (`short_name`), пустое → фолбэк на обычное название. Тогда редактор правит имя один
раз, и оно меняется на всех страницах сразу. Если привязать имя к странице, одна сущность окажется
названа по-разному на пяти страницах — и никто не будет знать, где правда.

**Блок «Где применяется» = выборка по связи**, а не рукописные плитки: у группы стоит галочка
«применимо тиснение», страница услуги сама собирает 6–9 плиток с `short_name`. Рукописный список
разъедется при первой правке каталога — и уже разъехался: до 17.07 в блоке лежал винегрет из
разделов, групп и отдельных товаров вперемешку (12 плиток, три уровня иерархии).

**Уровень для таких блоков — группа товаров.** Раздел врёт (тиснение не применяется ко всей
«Широкоформатной печати»), товар слишком узок («3D стикеры со смолой») и нестабилен. Группа — тот
уровень, на котором клиент принимает решение.

### Лимиты

Копим по мере вёрстки: в админке нужны лимиты, иначе длинный текст ломает раскладку.

| Блок | Поле | Лимит |
|---|---|---|
| Виды тиснения | заголовок | ≤ 30 знаков |
| Виды тиснения | описание | ≤ 130 знаков |
| Заголовок секции | h2 | ≤ 27ch (иначе упирается в край на 390) |
| Где применяется | подпись плитки | **≤ 20 знаков** (при 3 в ряд это 2 строки на 390) |

**«Где применяется» — живой пример, почему лимит нужен.** «Многостраничная полиграфия» и «Проектная
документация» просят третью строку даже при кегле 13: слово «Многостраничная» = 118px при доступных
87. Сейчас им разрешено вырасти — их ряд сетки на 18px выше остальных. Сократит редактор
(«Многостраничная», «Документация») — сетка станет ровной сама.

**Обрезка многоточием — не универсальная страховка.** `-webkit-line-clamp` **не работает на
флекс-элементе**: флекс блокифицирует `display`, `-webkit-box` превращается в блок, clamp молча
отключается, и текст режется по `overflow:hidden` насухо, без многоточия. Проверять замером
(`scrollHeight` против `clientHeight`), а не наличием правила в CSS.

Не покрыто, но нужно: карточки портфолио (тег + заголовок), строки цен, чек-лист требований.
Альтернатива лимиту — фиксированная высота карточки + многоточие; выбирать осознанно, а не по факту.
