## 1. Цвет — роли, а не просто значения

Сейчас есть только `Black` и `Brand`. Для иерархии и читабельности нужны **роли текста и
поверхностей**. Предлагаю (значения подбираются под твой `#222220`):

### Текст
| Токен | Значение | Где |
|---|---|---|
| `text/primary` | `#222220` | заголовки, основной текст |
| `text/secondary` | `#6E6E6A` | подписи, второстепенное, лиды под заголовком |
| `text/tertiary` | `#9A9A96` | плейсхолдеры, мета, disabled-подписи |
| `text/onBrand` | `#222220` | текст на жёлтой кнопке (чёрный, не белый — контраст) |
| `text/onDark` | `#FFFFFF` | текст на тёмном футере/CTA |
| `text/link` | `#222220` + underline на hover | ссылки в тексте |

### Поверхности и линии
| Токен | Значение | Где |
|---|---|---|
| `surface/page` | `#FFFFFF` | фон страницы |
| `surface/subtle` | `#F6F6F4` | подложки секций, пустые состояния |
| `surface/card` | `#FFFFFF` | карточки |
| `surface/dark` | `#222220` | футер, финальный CTA |
| `border/default` | `#E4E4E4` | рамки карточек, поля |
| `border/strong` | `#CFCFCB` | hover/focus рамок |

### Бренд (один акцент — дисциплина!)
| Токен | Значение | Где |
|---|---|---|
| `brand/base` | `#f7a705` | главный акцент, primary-кнопка, активный таб |
| `brand/hover` | `#e09905` | hover primary |
| `brand/soft` | `#FEF3D9` | подложки бейджей/выделений |

Семантика (статусы заказа, ошибки форм) — отдельно: `success / warning / danger / info`.
Их в ките сейчас нет, а формы и корзина без них не живут — добавить.

> **Правило одного акцента:** жёлтый — только для главного действия и активного состояния.
> Если жёлтым выделено всё — не выделено ничего. Это половина проблемы «нет иерархии».

## 2. Типографика — шкала вместо двух размеров

Заголовки — `AttractiveCond` (узкий, КАПС, как у тебя сейчас в секциях). Текст/UI — `Roboto`.
Главный фикс читабельности: **поднять line-height текста с 1.12 до 1.4–1.5**.

> **Шрифт заголовков (решение 2026-06-16):** `AttractiveCond` недоступен в среде Figma MCP
> («font family does not exist» — самого файла шрифта нет, есть только ссылки в старых стилях).
> Поэтому заголовки v1 реализованы на **`Roboto Condensed`** (узкое начертание, одна семья с
> текстовым Roboto → цельная типосистема). Display — Condensed Black, H1–H3 — Condensed Bold,
> H4 — Condensed SemiBold. Старые AttractiveCond-стили перенесены в папку `_legacy/` (не удалены,
> макеты не сломаны). Размеры/интерлиньяж/трекинг — по таблице ниже.

### Заголовки (Roboto Condensed, UPPERCASE)
| Стиль | Size / Line | Tracking | Где |
|---|---|---|---|
| `Display` | 44 / 48 | +1 | H1 крупных лендингов (главная) |
| `H1` | 34 / 38 | +1 | заголовок страницы (название технологии) |
| `H2` | 26 / 30 | +1.5 | заголовки секций (ОПИСАНИЕ, ПОРТФОЛИО) |
| `H3` | 20 / 24 | +2 | подзаголовки внутри секций |
| `H4` | 16 / 20 | +2 | мелкие заголовки, шапки карточек |

> Текущий `H2 20/24 tracking 3` остаётся как `H3`. Большой tracking (3) оставляем только
> для мелких КАПС-меток, на крупных заголовках он разваливает слово — уменьшил до +1…+1.5.

### Текст и UI (Roboto)
| Стиль | Size / Line | Weight | Где |
|---|---|---|---|
| `Lead` | 18 / 27 | 400 | вводный абзац под H1 |
| `Body` | 16 / 24 | 400 | основной текст (**было 16/1.12 → стало 16/24**) |
| `Body·strong` | 16 / 24 | 600 | акценты в тексте |
| `Small` | 14 / 20 | 400 | подписи, таблицы спеков, вторичное |

Правило: **на странице один H1**, секции — H2, дальше не прыгать через уровень.

## 3. Отступы и ритм — достроить шкалу

Сейчас: `2 / 4 / 8 / 12 / 16`. Обрывается — секциям нечем дышать. Достраиваю 8-пойнт-ритм:

`2 · 4 · 8 · 12 · 16 · 24 · 32 · 48 · 64 · 96`

Применение (рекомендация):
- внутри компонента (иконка↔текст): `4 / 8`
- между полями формы: `8` (твоё правило из доки — оставляем)
- паддинг карточек/модалок: `16` (твоё правило — оставляем)
- между элементами блока: `12 / 16`
- между подсекциями: `24 / 32`
- **между крупными секциями страницы: `64`** (сейчас секции слипаются — это главный фикс «структуры»)
- контейнер контента: max-width `1156` (твоё правило из доки), сетка 12 колонок, gutter `24`

## 4. Кнопки — три размера, фиксированные

Сейчас кнопка одна → размеры гуляют. Стандарт (высота фиксирована, ширина по контенту):

| Размер | Высота | Паддинг X | Текст | Где |
|---|---|---|---|---|
| `S` | 36 | 12 | Button 14 | компактные действия, в карточках |
| `M` | 44 | 16 | Button 15 | дефолт (**44 = минимальная тач-зона, твой стандарт из мобилки**) |
| `L` | 52 | 20 | Button 16 | главный CTA на странице |

Варианты (один и тот же атом, меняется только заливка):
- `primary` — `brand/base`, текст `text/onBrand` (чёрный). Главное действие, **один на экран-зону**.
- `secondary` — белый фон, `border/default`, текст primary. Вторичные действия.
- `ghost` — без фона/рамки, текст primary. Третичные, ссылки-действия.
- `onDark` — для тёмного CTA/футера.

Состояния обязательны для каждого: `default / hover / pressed / disabled / focus`.
Radius кнопки: `8` (между inside 4 и outside 14 — кнопка не «поле» и не «карточка»).

> Бейдж «−5%» на кнопке расчёта — отдельный атом `badge`, не вшивать в текст.

## 5. Стандарты компонентов (структура анатомии)

Единые правила, чтобы компоненты не «разъезжались»:
- **Карточка** = media (4:3 или 1:1) + body(padding 16) + footer(action). Radius `14`, border `default`.
- **Модалка** = Head (title + close) / Body / Footer с дивайдерами, padding 16 — **твоё правило, оставляем**.
  Ширины из доки: `1156 / 764 / 520 / 380`. Зафиксировать как enum, не плодить произвольные.
- **Таб** (для каталога технологий/материалов): высота 44, активный — `brand/base` подчёркивание
  или заливка `brand/soft`; неактивный — `text/secondary`. Контейнер скроллится по горизонтали.
- **Chip** (есть в ките) — для «сочетается с», фильтров. Не путать с кнопкой.
- **Поле ввода** — высота 44, radius `4`, border `default`, focus — `border/strong`. min-width 170 (твоё правило).

Все компоненты именовать по схеме `Component/Variant=…/Size=…/State=…` (см. §7).

## 6. Иерархия и читабельность — правила страницы

Чек-лист, по которому ловить «всё одинаковое»:
1. **Один H1 на страницу.** Дальше — только вниз по уровням (H2 → H3), без прыжков.
2. **Контраст размером, весом и цветом** — минимум 2 из 3 между уровнями
   (заголовок крупнее + жирнее + темнее, чем подпись).
3. **Вторичное — серым** (`text/secondary`), не чёрным. Если всё чёрное — нет глубины.
4. **Один акцент на зону** — жёлтым выделено максимум одно главное действие в видимой области.
5. **Воздух между секциями 64** — глаз должен видеть границы блоков без линий.
6. **Длина строки текста** — не шире ~75 символов (≈680px при 16px). Иначе абзацы не читаются.
7. **Тач-зоны ≥ 44px** для всех кликабельных (твой стандарт — распространить на десктоп-кнопки тоже).

## 7. Структура Figma-файла и хендофф

То, чего, по твоим словам, нет. Предлагаю навести порядок:

### Организация документа (страницы Figma)
```
📄 0 · Cover            — превью, версия, ченджлог
📄 1 · Foundations      — цвета, типографика, отступы, радиусы, эффекты (только переменные/стили)
📄 2 · Components        — библиотека компонентов (по группам: Actions / Inputs / Cards / Nav / Overlays)
📄 3 · Patterns          — собранные паттерны (hero, секция-портфолио, FAQ-блок)
📄 4 · Pages · Web        — макеты десктоп
📄 5 · Pages · Mobile     — макеты адаптив
📄 6 · Docs               — правила, ТЗ к контенту (у тебя уже хорошо проработано)
🗄 _Archive               — старые итерации (убрать из рабочих страниц, чтобы не путали)
```
Сейчас компоненты, макеты и доки лежат на одном холсте «Веб» (68000px шириной) — отсюда «нет
структуры». Разнести по страницам. Дубли итераций («Каталог технологии» ×2, «Материал/Технология»
×2) — оставить одну актуальную, остальное в `_Archive`.

### Переменные (Variables)
- Коллекция `Primitives` (сырые значения) → `Semantic` (роли из §1) → биндить компоненты к `Semantic`.
- Режим пока один — `Light`. Тёмная версия **не планируется** сейчас. Но архитектуру semantic-ролей
  держим mode-ready: когда Dark понадобится, это будет добавление режима в коллекцию, а не переделка.

### Хендофф (передача в разработку)
Чек-лист готовности макета к отдаче:
- [ ] Все цвета/тексты/отступы — **из переменных и стилей**, без ручных значений (hardcode).
- [ ] Компоненты — инстансы из библиотеки, без отвязанных копий.
- [ ] Auto-layout на всех контейнерах (чтобы dev видел паддинги/гэпы, а не координаты).
- [ ] Состояния интерактива нарисованы (hover/active/disabled/empty/error).
- [ ] Адаптив: ключевые брейкпоинты (≥1396 / 1156 / 768 / 390) — твои правила каталога уже есть.
- [ ] Назван слой-контракт: фреймы названы по смыслу (Hero / Specs / Gallery), не «Frame 427319581».
- [ ] Сетка и контейнер (max 1156) включены и видимы.
- [ ] Аннотации: где CMS-поле, где статика, какие лимиты текста (у тебя ТЗ к контенту — уже сильная база).
- [ ] Ссылка на этот `DESIGN_SYSTEM.md` или его Figma-аналог приложена.
