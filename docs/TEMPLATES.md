# Карта шаблона страницы каталога (для переноса на WordPress)

**Вывод сразу: отдельный шаблон материалам не нужен.** Из 13 блоков общие 11 — различий на два
слота. Отдельная вёрстка = третья копия, которая разъедется с первой правкой (это уже произошло
внутри одной страницы: одна сущность была названа по-разному, см. `DESIGN_SYSTEM.md` §10).

**Правильно:** один гибкий шаблон «страница каталога» на ДВА **типа записи** — `технология`
(= `услуга`, название пока не решено) и `материал`. Блоки включаются флагами, а «Требования
к макету» и «Характеристики» — два варианта одного слота. Общее остаётся общим и правится один раз.

Эталон вёрстки: `prototypes/usluga.html` (тип «технология», пример — тиснение фольгой).

**Комплект документов переноса:** `HANDOFF.md` (разработчику: порядок работ, интеграции, приёмка) ·
**этот файл** (карта блоков) · `CMS_FIELDS.md` (админка: поля, лимиты, крайние значения) ·
`QUESTIONS.md` (вопросы до старта, по адресатам) · `DESIGN_SYSTEM.md` (правила, источник правды —
код) · `IA_catalog.md` (архитектура каталога).

## 2. Поля и кто ими управляет

Правило из `DESIGN_SYSTEM.md` §10: **ни один блок не уходит в разработку без ответа «кто и как его
администрирует»**. Здесь этот ответ.

| Блок | Источник данных | Поля | Лимиты |
|---|---|---|---|
| Герой | запись | H1, лид, буллеты (3–5), кадр | H1 ≤ 27ch (иначе упирается в край на 390) |
| Статбар | запись | 4 пары «цифра + подпись» | цифра ≤ 6 знаков |
| Преимущества | запись | 4 плитки: иконка, заголовок, текст | заголовок ≤ 30 |
| Виды | запись, повторитель | заголовок, описание, кадр | заголовок ≤ 30, описание ≤ 130 |
| Цены | запись, повторитель | название, срок, цена | название ≤ 24 |
| Калькулятор | ссылка с предустановкой | заголовок, подпись, параметр (`тиснение` / `бумага X`) | — |
| Портфолио | **выборка по связи** | работы с флагом «показывать здесь» | подпись ≤ 24 |
| Требования | запись + медиатека | чек-лист (4), файлы (PDF/AI) | пункт ≤ 40 |
| Характеристики | **свойства материала** | таблица «параметр → значение» | значение ≤ 20 |
| ЧАВО | запись, повторитель | вопрос, ответ | вопрос ≤ 60 |
| Где применяется | **выборка по связи** | группы с флагом «применимо» | подпись = `short_name` ≤ 20 |
| С чем сочетается | **выборка по связи** | технологии/отделки | заголовок ≤ 20, подпись ≤ 40 |
| Отзывы | виджет (внешний) | — | — |
| Статьи | **выборка по рубрике** | — | заголовок ≤ 60 |
| Футер, шапка | глобально | — | — |

**Выборка по связи, а не рукописные плитки.** У группы каталога стоит галочка «применимо тиснение»
— страница собирает блок сама. Рукописный список разъезжается при первой правке каталога: до 17.07
в «Где применяется» лежал винегрет из разделов, групп и товаров вперемешку (12 плиток, три уровня
иерархии сразу).

**Два имени у сущности** (см. §10 ДС): навигационное — в меню и URL, клиентское `short_name` (≤20) —
в плитках и тексте. Клиентское живёт **у самой сущности**, не у страницы, иначе одна вещь окажется
названа по-разному на пяти страницах.

## 2.1. Мы не продаём технологию и не продаём материал

Ключевое (уточнено 17.07, я понял неверно и заложил в карту лишний вопрос про «калькулятор
листов»): **продаётся всегда изделие.** Тиснение и бумага — это его свойства.

Отсюда два следствия для разработки:

1. **Калькулятор один на сайт.** Он про продукт: взял визитки → добавил тиснение или выбрал бумагу →
   увидел цену. Страница услуги или материала — не своя витрина, а **вход в общий калькулятор с
   предустановкой**: ссылка передаёт параметр (`тиснение`, `бумага X`). Ни второго компонента, ни
   флага не нужно.
2. **Блок «Цены» у всех типов — тоже про продукт.** «Визитки с тиснением от 18 ₽», «Обложки
   каталогов от 60 ₽»; у материала будет «Визитки на дизайнерской бумаге от X ₽». Строка прайса =
   изделие + свойство, а не свойство само по себе.

Это же объясняет, почему «Где применяется» = группы товаров (см. §10 ДС): страница свойства
существует, чтобы привести к изделию, на котором это свойство продаётся.

## 3. Что уже есть на живом сайте

Страницы услуг и продукции **существуют** — их переделывают ради дизайна, а не создают с нуля.
Значит перенос — это:

1. завести типы записей и поля по таблице §2;
2. перевести существующий контент в поля (важно: **лимиты**, иначе сетки поедут — см. §10 ДС);
3. заменить шаблон вывода на вёрстку прототипа;
4. подключить `assets/css/{tokens,kit,fonts}.css` — **с `?v=`**, иначе браузеры будут отдавать
   старый CSS часами (ловили 16.07: телефон показывал новый HTML со старым CSS).
